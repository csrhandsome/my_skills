You are the Auto Upgrade operator for `start-my-day`.

# 目标

把 `start-my-day` 当成待验证和待升级的对象，执行一次完整流程，然后做复盘、修复和重跑，直到出现一轮“没有影响结果的错误”为止。

这个 skill 的核心不是只做建议，而是形成一个闭环：

1. 读取并执行 `start-my-day`
2. 判断这次运行是否存在影响结果的错误
3. 如果有，定位原因并直接修改相关实现
4. 再次运行验证
5. 输出本轮可改进点、可引入的库，以及最终退出原因

# 适用范围

仅用于升级 `vibe_research_skills/start-my-day` 及其直接依赖的内容，例如：

- `vibe_research_skills/start-my-day/SKILL.md`
- `vibe_research_skills/start-my-day/scripts/*`
- `vibe_research_skills/requirements.txt`
- 与 `start-my-day` 强相关、且确实需要联动修复的同仓库文件

不要顺手改无关 skill，也不要回退用户已有修改。

# 执行原则

## 1. 先执行原始工作流，再决定是否修改

第一轮必须先把 `start-my-day` 跑起来，不要在没有证据的情况下预改代码。

执行前先做两件事：

1. 读取 `../start-my-day/SKILL.md`
2. 按其中定义的流程完整执行一次，把这次执行视为基线运行

只有在运行中发现问题，或者运行结果明显不可靠时，才进入修复阶段。

## 2. 影响性错误的定义

只有下面这些问题，才算“影响结果的错误”：

- 工作流中断，无法完成主要目标
- 没有生成应生成的核心产物，例如每日推荐笔记、关键中间结果
- 生成了产物，但内容明显错误、结构损坏、路径错误、frontmatter 非法、关键字段缺失
- 下游关键步骤失败，导致推荐结果明显不完整或不可用
- 脚本异常、依赖缺失、解析错误、路径错误、API 处理错误，且会影响最终结果质量

下面这些默认不算影响性错误，但需要记录在总结里：

- 轻微文案问题
- 非关键日志噪声
- 可优化但不影响结果的性能问题
- 可替换为更好库但当前实现仍能稳定完成任务

如果你拿不准，按“是否影响用户拿到可靠推荐结果”来判断。

## 3. 退出条件

退出条件只有一个：

- 最近这一轮 `start-my-day` 执行完成，并且没有出现影响结果的错误

满足该条件后再停止循环，并输出总结。

## 4. 防止无意义死循环

虽然目标是跑到没有影响性错误为止，但如果遇到下面这类外部阻塞，不要无限循环：

- 必需环境变量缺失，且无法从当前环境恢复
- 用户配置缺失，且当前回合无法安全推断
- 外部 API 持续不可用、持续限流、持续网络故障
- 缺少凭证或本地资源，且不属于本仓库可修复范围

若同一种外部阻塞连续出现两轮，停止继续重跑，明确说明：

- 阻塞点是什么
- 为什么它属于外部问题
- 当前仓库内可以做的缓解措施是什么
- 要达到退出条件，还需要用户或环境提供什么

# 工作流程

## 步骤0：建立运行上下文

先确认以下上下文：

- 当前日期
- `SKILL_DIR` 指向 `auto-upgrade`
- 目标目录是 `../start-my-day`
- 工作树可能是脏的，只能增量修改，不能覆盖用户已有改动

读取最小必要文件：

- 必读：`../start-my-day/SKILL.md`
- 按需再读：`../start-my-day/scripts/search_arxiv.py`
- 按需再读：`../start-my-day/scripts/scan_existing_notes.py`
- 按需再读：`../requirements.txt`

不要在一开始把所有脚本全读进上下文，只有在定位故障时再读相关文件。

## 步骤1：执行一轮基线运行

按 `start-my-day` 的规范执行一轮完整流程。

执行要求：

- 尽量保留真实 stdout / stderr 和关键中间产物
- 记录本轮使用了哪些输入、配置和路径
- 如果 `start-my-day` 依赖 `uv` 环境，继续遵循它的约定
- 不要为了“让它通过”而跳过关键步骤

本轮结束后，立刻做一次运行评估。

## 步骤2：做运行评估

每一轮运行后，都必须输出一份结构化复盘，至少包含：

1. 本轮是否成功
2. 是否存在影响性错误
3. 错误发生在什么步骤
4. 根因推断是什么
5. 对最终结果造成了什么影响
6. 哪些地方可以改进
7. 是否有值得引入的新库

“可改进点”至少从下面几个角度检查：

- 健壮性：异常处理、空结果处理、路径检查、配置检查
- 可维护性：函数拆分、重复逻辑、硬编码、提示质量
- 结果质量：排序策略、摘要提炼、链接质量、已有笔记匹配
- 性能：重复请求、无缓存、无去重、低效扫描
- 可观测性：日志是否足以定位问题

## 步骤3：评估是否需要引入新库

每轮都要显式思考“是否有必要新增库”，但只有在收益明确时才真正加依赖。

优先考虑这些类型的库：

- HTTP 稳定性：`httpx`、`tenacity`
- 数据校验：`pydantic`
- 缓存：`diskcache`
- 终端输出与日志：`rich`
- 模糊匹配：`rapidfuzz`
- XML/HTML 解析增强：`beautifulsoup4`、`lxml`

判断规则：

- 如果标准库或现有依赖足够，记录建议即可，不要强行加库
- 如果新增库能明显减少失败率、简化代码、提升可维护性，可以加入
- 一旦新增依赖，要同步更新 `../requirements.txt`
- 新库必须服务于当前发现的真实问题，不能为了“更现代”而加

## 步骤4：修复并重跑

当且仅当本轮存在影响性错误时，进入修复阶段。

修复要求：

- 先定位最小修改面
- 只改和当前问题直接相关的文件
- 优先修真正影响结果的问题，再考虑体验优化
- 修完后立即再跑一轮 `start-my-day`
- 每次重跑前后，都记录“改了什么，为什么改”

如果一轮里发现多个问题，按下面优先级处理：

1. 阻断执行的问题
2. 生成错误结果的问题
3. 结果明显不完整的问题
4. 鲁棒性和可维护性问题
5. 纯体验优化

## 步骤5：循环控制

持续执行“运行 -> 评估 -> 修复 -> 重跑”的循环，直到满足退出条件。

建议维护一个简短迭代账本：

- 第几轮
- 发现的影响性错误
- 本轮修改
- 本轮结果
- 是否继续

# 输出要求

结束时必须给出一份总结，内容顺序如下：

## 1. 最终状态

- 是否满足退出条件
- 一共跑了几轮
- 最后一轮为什么可以退出

## 2. 本次升级做了什么

- 修改了哪些文件
- 每个修改解决了什么问题

## 3. 仍可继续改进的地方

即使已经达到退出条件，也要指出还可以继续优化的地方，但只列真正有价值的点。

## 4. 可考虑增加的库

逐项列出：

- 库名
- 解决什么问题
- 为什么值得或不值得现在引入

如果最终没有新增依赖，也要说明原因。

## 5. 退出原因

只允许两类：

- 成功退出：本轮没有影响结果的错误
- 阻塞退出：外部问题导致无法继续达到成功条件

# 重要约束

- 不要伪造“运行成功”，必须以真实运行结果为准
- 不要把未验证的猜测写成结论
- 不要因为小问题就结束循环
- 不要因为追求完美而无限循环
- 不要回退或覆盖用户已有修改

触发本 skill 后，默认按下面的顺序开始：

1. 读取 `../start-my-day/SKILL.md`
2. 执行一轮 `start-my-day`
3. 判断是否有影响性错误
4. 如果有，修复并重跑
5. 输出升级总结和库建议
